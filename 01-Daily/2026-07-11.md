# 2026-07-11 — Session: Hyprwallpaper + Multi-Monitor Fix

## Contexto

El usuario reportó que después de que la PC se apagó por batería, el wallpaper cambió al default de Hyprland y las ventanas fallan al conectar pantalla externa.

## Investigación realizada

### Hallazgo 1: Dual daemon conflict
- Tanto `awww-daemon` (PID 1996) como `hyprpaper` (PID 79068) estaban corriendo compitiendo por las mismas capas de fondo en ambos monitores
- `awww` es el daemon real (maneja cycling via `wallpaper.sh` usando `awww img`)
- `hyprpaper` era redundante y estaba encima cubriendo el wallpaper de awww

### Hallazgo 2: hyprpaper format break
- hyprpaper se actualizó de v0.8.4-2.1 → v0.8.4-3.1 el 28 de junio
- El formato viejo `wallpaper = , /path` dejó de funcionar
- Desde el 28 de junio, hyprpaper fallaba silenciosamente con "Monitor has no target"
- **No fue causado por la muerte de batería** — fue por la actualización

### Hallazgo 3: La intervención anterior empeoró el problema
- En una sesión anterior se editó hyprpaper.conf con formato de bloques
- Esto hizo que hyprpaper cargara `wallpaper.jpg` estático, cubriendo el wallpaper de awww
- Antes de esa edición, las capas de hyprpaper estaban vacías/transparentes y awww se veía

## Acciones tomadas

### ✅ Fase 1: Deshabilitar hyprpaper
```bash
systemctl --user stop hyprpaper && systemctl --user disable hyprpaper
```
- Servicio removido de `graphical-session.target.wants`
- Config respaldada: `mv ~/.config/hypr/hyprpaper.conf ~/.config/hypr/hyprpaper.conf.bak`
- **Resultado:** Solo quedan capas de awww-daemon en ambos monitores (0 capas hyprpaper)

### ⚠️ Fase 2: Keybinds multi-monitor (parcial)
Se agregaron 4 keybinds a `~/.config/hypr/hyprland.lua`:
- `SUPER + [ / ]` — foco entre monitores
- `SUPER + SHIFT + [ / ]` — mover ventana al otro monitor

**Primer intento (FALLÓ):**
```lua
hl.bind(mainMod .. " + bracketleft",  hl.dsp.focus({ monitor = "previous" }))
hl.bind(mainMod .. " + bracketright", hl.dsp.focus({ monitor = "next" }))
```
- `previous`/`next` no son valores válidos para el parámetro `monitor`
- Los dispatchers se registraron como `__lua` pero no hacían nada

**Segundo intento — bug encontrado y corregido (2026-07-11, sesión revisión):**

El SHIFT-bind tenía bug de orden: `focus()` cambiaba de monitor ANTES de `window.move()`, así que movía la ventana ya enfocada en el monitor destino (no la original). Además `workspace = "+0"` es offset relativo 0 — no traslada nada entre monitores.

Fix: reemplazado por dispatcher nativo `movewindow` con sintaxis `mon:+1`/`mon:-1` (confirmado en binario de Hyprland vía `strings`), que mueve la ventana enfocada directo al monitor relativo en un solo paso:

```lua
hl.bind(mainMod .. " + bracketleft",  hl.dsp.focus({ monitor = "-1" }))
hl.bind(mainMod .. " + bracketright", hl.dsp.focus({ monitor = "+1" }))
hl.bind(mainMod .. " + SHIFT + bracketleft",  hl.dsp.exec_cmd("hyprctl dispatch movewindow mon:-1"))
hl.bind(mainMod .. " + SHIFT + bracketright", hl.dsp.exec_cmd("hyprctl dispatch movewindow mon:+1"))
```
- Config recargada con `hyprctl reload` → OK
- **⚠️ AÚN NO VERIFICADO EN VIVO** — requiere pantalla externa conectada (ahora mismo solo eDP-1 activo) para probar `SUPER+]` y `SUPER+SHIFT+]`

## Nota técnica

En Hyprland 0.55.4 con config Lua, `hyprctl dispatch` parsea argumentos como expresiones Lua. Dispatchers como `focusmonitor`, `cyclenext`, `movefocus` fallan con errores de parsing. La API Lua expone `hl.dsp.focus({ monitor })` con valores `+1`/`-1` para relativo, o nombres de monitor directos. No hay `window.move({ monitor })` — solo `window.move({ workspace })`.
