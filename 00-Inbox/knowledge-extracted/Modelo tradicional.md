## Diapositiva 5

Modelo Cascada
Es el paradigma más antiguo de la ingeniería de software.
Se aplica cuando:
El problema se entiende bien
En caso de mejoras, cuando los requerimientos estén bien definidos
Problemas:
Rara vez se sigue el flujo de trabajo secuencial.
Es difícil para el cliente indicar todos los requerimientos.
El cliente debe ser paciente, ya que no habrá una versión hasta las etapas avanzadas.
Errores graves no pueden detectarse hasta que se revisa el programa funcional

## Diapositiva 7

Prototipos de modelos de proceso
Se puede ocupar como un modelo de proceso independiente.
Sirve para identificar los requerimientos del software.
Problemas:
Los interesados pueden ver el prototipo como una versión funcional.
Se pierde de vista la calidad y la capacidad de mantenimiento del software.
Ser tentado a ceder en algunos aspectos de la implementación, para lograr que el prototipo funcione rápidamente.

## Diapositiva 9

Modelo de espiral
Es un modelo de proceso de software evolutivo que acopla la naturaleza iterativa de los prototipos con los aspectos controlados del modelo cascada.
Problemas:
Exige una considerable experiencia en  evaluación del riesgo y depende de esta experiencia para tener éxito.
