# Comunicación de Requerimientos

Cómo obtener, documentar y validar requerimientos de stakeholders.

## 10 Principios de Comunicación

1. **Escuchar** — no asumir, preguntar y escuchar activamente
2. **Prepararse** — investigar el dominio antes de la reunión
3. **Facilitar** — guiar la conversación, no dominarla
4. **Cara a cara** — preferido sobre email/documentos
5. **Documentar** — registrar acuerdos inmediatamente
6. **Colaborar** — trabajar juntos, no en silos
7. **Mantener enfoque** — evitar deriva de alcance
8. **Dibujar** — diagramas y sketches comunican mejor que texto
9. **Acordar/moverse** — resolver desacuerdos rápidamente
10. **Negociación ganar-ganar** — todos ganan, no hay perdedores

## Identificación de Stakeholders

| Tipo | Rol | Interés |
|------|-----|---------|
| **Usuarios finales** | Usan el sistema diariamente | Usabilidad, eficiencia |
| **Clientes/Sponsors** | Financian el proyecto | ROI, alcance, plazos |
| **Desarrolladores** | Construyen el sistema | Viabilidad técnica |
| **Gestores** | Supervisan el proyecto | Cumplimiento de objetivos |
| **Reguladores** | Definen normas | Cumplimiento legal |

## Tipos de Stakeholders según Requerimientos

- **Los que saben qué quieren** → extraer y documentar
- **Los que no saben qué quieren** → facilitar, mostrar prototipos
- **Los que quieren todo** → gestionar alcance con MoSCoW
- **Los que cambian constantemente** → control de cambios formal

## Escenarios de Entrevista

| Escenario | Estrategia |
|-----------|-----------|
| Cliente pasivo | Facilitar con preguntas abiertas, mostrar opciones |
| Cliente exigente | Gestionar expectativas, documentar alcance |
| Cliente cambia todo | Control de cambios, cada cambio tiene impacto |

## Estructura de Preguntas Iniciales

1. ¿Quién usará el sistema?
2. ¿Qué problemas resuelve?
3. ¿Cómo funciona hoy (sin software)?
4. ¿Cuáles son las restricciones?
5. ¿Cómo mediremos el éxito?
